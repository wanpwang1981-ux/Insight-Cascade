# 系統架構分析 (System Architecture Analysis)

本文檔旨在詳細闡述 AI 協作對話系統的設計與架構，作為開發階段的最高指導原則。

## 1. 需求分析 (Requirement Analysis)

### 1.1. 功能性需求 (Functional Requirements)

-   **F-1: 模組化角色管理**
    -   **F-1.1:** 系統應允許使用者建立、設定和儲存無限數量的 AI 聊天模組（角色）。
    -   **F-1.2:** 每個模組應包含以下可配置屬性：
        -   `module_id`: 唯一的識別碼。
        -   `character_setting`: 角色的詳細設定（個性、背景、知識領域、語氣等）。
        -   `model_endpoint`: 使用的 AI 模型接口（例如：本地模型的路徑、Gemini API 的 URL）。
        -   `api_key`: (可選) 對應模型的 API 金鑰。
-   **F-2: 對話流程控制**
    -   **F-2.1:** 系統應能接收使用者的初始輸入（主題）。
    -   **F-2.2:** 系統需要一個「對話管理器」(Orchestrator) 來啟動和控制對話流程。
    -   **F-2.3:** AI 模組在生成回覆後，能夠建議下一個接手的模組 `module_id`。
    -   **F-2.4:** 對話管理器根據建議，將對話歷史和當前情境傳遞給下一個指定的模組。
-   **F-3: 同步等候機制**
    -   **F-3.1:** 在任一模組正在生成回覆時，系統應處於鎖定或等候狀態，防止其他模組同時觸發。
-   **F-4: 模組單元測試**
    -   **F-4.1:** 系統應提供一個獨立的測試介面或稿本，讓使用者可以與單一模組進行對話，以驗證其角色設定是否符合預期。

### 1.2. 非功能性需求 (Non-Functional Requirements)

-   **NF-1: 可擴展性 (Scalability)**: 系統架構應易於水平擴展，以支援未來更多的 AI 模組和更高的請求量。
-   **NF-2: 可維護性 (Maintainability)**: 程式碼應模組化、文件清晰、易於理解和修改。
-   **NF-3: 靈活性 (Flexibility)**: 更換或新增 AI 模型供應商（如從 Gemini 切換到 OpenAI）的過程應盡可能簡單。

## 2. 高階架構設計 (High-Level Architecture)

系統將採用微服務或模組化架構，主要由以下幾個核心元件組成：

```
+------------------+      +----------------------+      +--------------------+
|   使用者介面     |----->|     對話管理器       |<---->|    模組設定資料庫    |
| (Web/CLI)        |      |   (Orchestrator)     |      | (JSON/DB)          |
+------------------+      +----------+-----------+      +--------------------+
                                     |
                                     | 1. 分派任務
                                     |
             +-----------------------+-----------------------+
             |                       |                       |
             v                       v                       v
+--------------------------+  +--------------------------+  +--------------------------+
|     聊天模組 A           |  |     聊天模組 B           |  |     聊天模組 C           |
| (Chat Module A)          |  | (Chat Module B)          |  | (Chat Module C)          |
|  - 角色設定 A            |  |  - 角色設定 B            |  |  - 角色設定 C            |
|  - 模型接口 (Gemini)     |  |  - 模型接口 (Local)      |  |  - 模型接口 (Other)      |
+------------+-------------+  +------------+-------------+  +------------+-------------+
             |                           |                           |
             +---------------------------+---------------------------+
                                         | 2. 呼叫外部 AI
                                         v
+---------------------------------------------------------------------------------+
|                               外部 AI 服務                                      |
| (Gemini API / Local LLM / ...)                                                  |
+---------------------------------------------------------------------------------+

```

-   **使用者介面 (UI)**: 負責接收使用者的輸入並顯示對話結果。初期可以是簡單的命令列介面 (CLI)。
-   **對話管理器 (Orchestrator)**: 系統的大腦。負責解析使用者輸入、管理對話狀態、根據 AI 模組的指示調度下一個對話者。
-   **模組設定資料庫**: 儲存所有 AI 聊天模組的設定檔。初期可以使用 JSON 檔案，未來可升級為資料庫。
-   **聊天模組 (Chat Module)**: 獨立的執行單元。每個模組都封裝了與特定 AI 模型互動的邏輯。它接收來自管理器的指令和對話歷史，生成提示 (Prompt)，呼叫 AI 模型，並將結果回傳給管理器。

## 3. AI 可視化流程圖 (AI-Readable Flowchart)

這是一個為 AI 設計的、基於文字的流程圖，描述了單次對話的完整生命週期。

```mermaid
graph TD
    A[Start] --> B{使用者輸入主題};
    B --> C[對話管理器: 初始化對話];
    C --> D{讀取所有模組設定};
    D --> E[對話管理器: 指派給起始模組 (例如：模組 A)];

    subgraph "對話循環 (Conversation Loop)"
        E --> F[模組 A: 接收對話歷史與任務];
        F --> G{模組 A: 根據角色設定和歷史，建構 Prompt};
        G --> H[模組 A: 呼叫外部 AI 模型 (e.g., Gemini)];
        H --> I{AI 模型回傳結果};
        I --> J[模組 A: 解析結果，並決定下一個模組];
        J --> K{回傳 (回覆內容, 下一個模組 ID) 給管理器};
    end

    K --> L[對話管理器: 接收結果];
    L --> M{更新完整對話歷史};
    M --> N{檢查下一個模組 ID 是否為 'End'};

    N -- No --> O[對話管理器: 指派給下一個模組];
    O --> F;

    N -- Yes --> P[對話管理器: 將最終結果呈現給使用者];
    P --> Q[End];

```

**流程圖解說:**

1.  **開始 (Start)**: 流程啟動。
2.  **使用者輸入**: 使用者提供一個對話主題或問題。
3.  **初始化**: `對話管理器` 建立一個新的對話 session，並從 `模組設定資料庫` 讀取所有可用的 AI 角色設定。
4.  **指派**: `對話管理器` 選擇一個預設的起始模組 (例如模組 A) 來開始對話。
5.  **進入對話循環**:
    *   **建構 Prompt**: 被指派的模組 (如模組 A) 會結合自身的 `角色設定` 和當前的 `對話歷史`，建立一個完整的 Prompt。
    *   **呼叫模型**: 模組 A 將建構好的 Prompt 發送給它所設定的 `模型接口` (例如 Gemini API)。
    *   **解析與決策**: 模組 A 收到 AI 模型的回覆後，會進行兩件事：
        1.  格式化要呈現給使用者的文字。
        2.  根據對話內容和內建邏輯，決定接下來應該由哪個模組接手 (或者結束 `End`)。
    *   **回傳**: 模組 A 將自己的回覆內容和下一個模組的 ID 回傳給 `對話管理器`。
6.  **更新與檢查**: `對話管理器` 收到回傳後，會將這次的回覆加入到完整的 `對話歷史` 中，並檢查下一個模組 ID。
7.  **循環或結束**:
    *   如果下一個模組 ID 不是 'End'，`對話管理器` 就會根據該 ID 指派新的模組，並重複**對話循環**。
    *   如果 ID 是 'End'，則代表對話結束。
8.  **結束 (End)**: `對話管理器` 將完整的對話結果輸出給使用者，流程結束。

## 4. 版本建置分析 (Build & Deployment)

-   **初期開發 (v0.1-v0.5)**:
    -   **環境**: Python 3.x。
    -   **依賴管理**: 使用 `requirements.txt`。
    -   **執行方式**: 本地命令列執行。
    -   **測試**: 手動執行單元測試稿本。
-   **中期發展 (v0.6-v1.0)**:
    -   **介面**: 考慮引入 Flask/FastAPI 建立簡單的 Web API。
    -   **部署**: 考慮使用 Docker 進行容器化，方便部署。
    -   **CI/CD**: (可選) 引入 GitHub Actions 自動化測試與建置。
-   **長期規劃**:
    -   **資料庫**: 將模組設定從 JSON 遷移至 SQLite 或更大型的資料庫。
    -   **前端**: 開發一個正式的前端介面 (例如使用 React/Vue)。
